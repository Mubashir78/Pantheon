# Composio MCP Server Bundle

Portable Composio MCP server config for Hermes Agent, designed to be installed via **Olympus UI V3 Onboarding**.

## What's in this bundle

```
composio-mcp.yaml     # MCP server config template (env var placeholders)
setup_credentials.py  # Helper Olympus backend calls to generate MCP credentials
README.md             # This file
```

## Env vars the onboarding flow must set

These are loaded from `~/.hermes/.env` and resolved by Hermes's built-in `_expand_env_vars()`:

| Variable | Description | How to get it |
|----------|-------------|---------------|
| `COMPOSIO_CONSUMER_KEY` | Consumer API key (`ck_...`) | From Composio API key or SDK session |
| `COMPOSIO_AUTH_TOKEN` | JWT Bearer token | Generated by Composio SDK on auth |

## How Olympus V3 onboarding should generate them

The user provides their Composio API key (`ak_...`) during onboarding. The Olympus backend generates the MCP credentials from it:

```python
from composio import Composio

# Authenticate with the user's API key
composio = Composio(api_key="ak_user_provided_key")

# The consumer key is typically available from the Composio client
# after authentication. The Auth token comes from the SDK session.
# See setup_credentials.py for the exact implementation.
```

## Config template

The `composio-mcp.yaml` uses `${VAR}` placeholders that Hermes's `_expand_env_vars()` resolves at config load time. The Olympus onboarding writes the resolved values into `~/.hermes/.env`.

## Installation

During Olympus V3 onboarding:
1. User signs up / logs in at [composio.dev](https://composio.dev) → gets API key
2. User pastes API key into Olympus onboarding UI
3. Olympus backend calls `setup_credentials.py` → generates consumer key + JWT
4. Olympus writes env vars to `~/.hermes/.env`
5. `composio-mcp.yaml` snippet is merged into `~/.hermes/config.yaml`

The MCP server is ready on next Hermes session start.
